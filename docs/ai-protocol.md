# AI 連携プロトコル

ステータス: draft / 最終更新: 2026-07-10

エージェント(Claude Code 等)がデッキを生成・編集するときの取り決め。語彙(何を書いてよいか)は [subset-spec.md](subset-spec.md) が定めるので、本書は**手順**(どう作業するか)と**やり取り**(人間がどう指示し、エージェントがどう報告するか)を定める。

## 1. 基本方針

1. **プロトコルを覚えるのはエージェントであって人間ではない。** 人間は自然言語で指示する。本書の規約はすべてエージェント側の行動規範として SKILL.md 経由で注入される。
2. **インターフェースはファイルと CLI のみ。** エージェントはソース `.tex` を直接編集し、検証は CLI で行う。エディタはファイル監視で追従するだけで、エージェントとエディタは直接通信しない。この分離により、チャットの置き場所(ターミナルの Claude Code、将来の内蔵チャットパネル)が変わってもプロトコルは不変。
3. **エージェントの検証手段は常に TeX。** HTML プレビューは人間用の近似であり、エージェントはそれを見ない。正しさは `deck lint`(語彙)と `deck check`(実コンパイル)で確認する。これにより、HTML プレビューが原理的に検出できない「行の溢れ・組版エラー」をエージェントが人間より先に潰せる。

## 2. エージェントの作業ループ

すべての編集タスクは次のループに従う。

```
1. deck outline <file>          構造を把握(全文を読まずにフレーム一覧を得る)
2. 対象フレームの行範囲だけ読む
3. 編集(Edit ツールでソースを書き換える)
4. deck format <file> --write   正規形に整える
5. deck lint <file>             エラー 0 になるまで修正
6. deck check <file>            内容量・レイアウトを変えた場合は実コンパイル検証
   (Overfull 警告・コンパイルエラーがフレームアドレス付きで返る)
7. deck snapshot --frame <addr> 見た目に確信が持てないときは画像で目視(vision)
8. 報告                          変更フレームのアドレス + 一行説明 + lint/check 結果
```

完了条件: フォーマット済み・lint エラー 0・(内容量を変えた場合)check 通過。これを満たすまで「完了」と報告しない。

## 3. アドレッシング(スライドの指し方)

人間・エージェント・CLI 出力の三者が共通で使うフレームの参照方法。

| 形式 | 例 | 用途 |
|---|---|---|
| 序数 | 「3枚目」/ `frame 3` | 会話での自然な参照(1 始まり、ファイル出現順) |
| タイトル | 「『実験結果』のフレーム」 | 並べ替えに強い。曖昧なら エージェントが候補を確認 |
| ラベル | `frame results`(`\begin{frame}[label=results]`) | 永続参照。lint 出力・スクリプトで使う |

- フレーム内の要素(「2番目の箇条書き」等)は自然言語でよい。エージェントが AST を介して解決する。機械可読な要素 ID は導入しない(会話の粒度では不要)。
- `deck outline` は 序数 / label / タイトル / 行範囲 / lint 指摘数 を一覧で返し(`--json` あり)、この対応表がアドレス解決の基盤になる。
- エディタはステータスバーに現在フレームのアドレスを表示し、クリックでコピーできる。「エディタで見ているこのスライド」をチャットに貼る橋渡し。

## 4. CLI 一覧(エージェント向け)

| コマンド | 用途 | 出力 |
|---|---|---|
| `deck outline <file>` | 構造把握・アドレス解決 | フレーム一覧(text / `--json`) |
| `deck lint <file>` | 語彙・規則の検証(L001〜) | ソース位置付き指摘(text / `--json`) |
| `deck format <file> --write` | 正規形化 | 差分の有無 |
| `deck check <file>` | 実コンパイルによる検証 | エラーと Overfull 警告をフレームアドレスに割り付けて報告 |
| `deck snapshot <file> --frame <addr> -o <png>` | フレームの見た目の自己確認 | 実コンパイル画像 |
| `deck export <file> -o <pdf>` | 最終出力 | PDF |
| `deck init` | 新規デッキプロジェクトの雛形生成(スキル同梱。§8) | 生成ファイル一覧 |

`deck check` はキャンバスフレーム([subset-spec.md](subset-spec.md) §2.8)について、savepos 実測による本文領域外へのはみ出し・オブジェクト重なりの検出も報告する(絶対配置は Overfull 警告が出ないため)。

`check` と `snapshot` は tectonic を使うため数秒かかるが、エージェントの検証は対話的操作ではないので許容する(人間のプレビューは常に即時の HTML 側)。

## 5. 行動規約(ガードレール)

SKILL.md に埋め込む規範。lint が機械的に検出できるものは括弧内の規則で裏打ちする。

1. ツール管理プリアンブルを編集しない(L006)。マクロ定義は `%% macros` 領域のみ(L003)。
2. サブセット語彙を優先する。生 LaTeX はサブセットで表現できない場合のみ使い、使ったことを報告に明記する(L001 で可視化される)。
3. 指示されていない変更をしない。特にフレームの順序・枚数・既存の文言は、指示の範囲外なら保持する。
4. 画像は `assets/` に置き、参照先の存在を確認する(L004)。
5. 新規デッキや大規模改編は、いきなり全文を書かずに**まずアウトライン(フレームタイトル一覧)をチャットで提案し、合意してから生成する**。アウトラインのレビューは安く、全文のレビューは高い。
6. git 管理下では 1 指示 = 1 コミットを目安にする。人間のレビューと巻き戻しの単位を指示単位に揃える。
7. 報告は「変更したフレームのアドレス+何をしたか一行」の列挙と、lint / check の結果で締める。人間はそれを見てエディタのプレビュー(自動反映済み)で確認する。

## 6. 人間側の指示パターン(例)

プロトコルではなく「こう話せば伝わる」という実例集。SKILL.md にも同梱する。

| パターン | 指示の例 | エージェントの動き |
|---|---|---|
| 叩き台生成 | 「この論文要旨から 15 枚で発表の叩き台を。実験結果に重心を」 | アウトライン提案 → 合意 → 生成 → lint/check → 報告 |
| 局所修正 | 「frame 7 の箇条書きを 3 点に圧縮して」 | 対象フレームのみ編集 → ループ 4〜8 |
| 横断修正 | 「全フレームのタイトルを体言止めに統一」 | outline で全体把握 → 一括編集 → 報告は変更一覧 |
| 検証駆動 | 「溢れてるフレームがないか確認して、あれば直して」 | `deck check` → Overfull のフレームを修正 → 再 check |
| 素材の取り込み | 「この図を『手法』のフレームに入れて」 | assets/ へ配置 → `\includegraphics` 挿入 → snapshot で目視確認 |
| スタイル合わせ | 「この会社テンプレ(pptx / 見本 PDF / スクショ)に合わせて」 | 見本から色・フォント・ロゴ・フッターを抽出 → `%% style` 領域に記述(語彙外の様式は preamble-extra へ。使ったことを報告に明記)→ snapshot と見本を並べて比較・反復([theme-design.md](theme-design.md) §3) |
| 見た目の微調整 | 「『結果』の表、窮屈なので余白を」 | §7 の微調整モード(対象固定 → 修正 → snapshot 前後比較 → 短い報告) |

## 7. 微調整モード

叩き台の生成だけでなく、仕上げの微調整も AI に依頼できる。通常タスクとの違いは、やり取りの粒度が細かく高速で、「窮屈」「バランスが悪い」のような視覚的判断を伴うこと。これを支える取り決めを定める。

### 依頼の形

エディタから依頼する場合、エディタが次の形のメッセージを合成してエージェントに送る。手打ちの自由文でも構わない(この書式は対象を確実に伝えるためのものであって、人間の義務ではない)。

```
[adjust]
file: slides/main.tex
target: frame 7(label=results)
selection: 122–128 行目
> (選択中のソースの引用)

指示: この表、窮屈なので余白をとって
```

対象 frame に `label` がない場合、エディタは依頼の合成時に label を自動付与してから target を記録する(序数アドレスは並べ替えで壊れるため。キャンバスフレームは L011 で label 必須)。

### エージェントの規則

1. **スコープ固定。** 変更は target のフレーム(選択があればその要素)に限る。対象外に手を入れるべきだと判断した場合(例: 内容を 2 枚に分けるべき)は、変更せずに提案として返す。
2. **見た目の依頼は視覚で検証する。** 外観に関する依頼では、修正の前後を `deck snapshot` で比較し `deck check` を通してから報告する。改善しなければ 2 回まで自律的に手直しし、それでも解決しなければ状況を報告して指示を仰ぐ。文言のみの修正(誤字等)は lint だけでよい。
3. **報告は最小限。** 高速な往復を妨げないよう「frame 7: 表の列間隔を拡大、フォントを \small に。check 通過」程度の 1〜2 行にする。
4. **文脈の継続。** 「もう少し」「やっぱり戻して」は直前の依頼と同じ target への追加依頼として解釈する。このためセッションは使い捨てにせず維持する(エディタ統合ではデッキごとに 1 本の永続セッションを保つ)。

### エディタ側の統合(段階的)

| 段階 | 形 |
|---|---|
| M2 時点 | エディタの隣(同一 VS Code ウィンドウ内)で Claude Code / Codex の公式拡張を開き、アドレスコピー(§3)で対象を貼って依頼する。他拡張の UI を内部 API で自動操作することはしない |
| Phase 9 | プレビュー・スライド一覧・ソースの選択から「AI に依頼」→ 依頼ボックス → エディタが `[adjust]` を合成し、provider-neutral な **AgentAdapter**(Claude Agent SDK / Codex SDK の別実装)の永続セッションへ送信。デッキごとに 1 セッション、書き込み可能な実行は同時に 1 本。セッション ID はコミットせずローカル保存 |

反映経路は二本立てとする([beamer-editor-additional-requirements.md](beamer-editor-additional-requirements.md) §4.6)。

- **外部 CLI(M2 形態)**: エージェントがライブファイルを直接編集し、エディタがファイル監視で取り込む。依頼前に保存し、dirty buffer があれば警告する。外部変更は undo 履歴の 1 ステップとして取り込み、競合時は黙って上書きせず diff を表示する。
- **内蔵 AI(Phase 9)**: ライブファイルを直接編集させない。依頼時に保存スナップショットと対象 frame の label + 内容ハッシュを記録し、エージェントは一時スナップショット(worktree)側を編集して `deck format` / `deck lint` / 必要に応じて `deck check` を一時側で実行する。完了時に対象 frame の差分を生成し、現在の frame ハッシュが依頼時と一致すれば **ShellHost 経由で 1 undo ステップとして自動適用**、人間が同じ frame を変更していた場合は自動適用せず差分レビューへ送る。

どちらの経路でも、気に入らない微調整はチャットで「戻して」と言っても、エディタで Cmd+Z しても戻せる(二重の安全弁)。内蔵 AI の実行中も人間は別の frame を編集できる(同じ frame を同時に編集した場合だけ明示的な競合になる)。

## 8. SKILL.md による配布

エージェントへの知識注入は Claude Code のスキル形式で行う。

```
skills/beamer-deck/
├── SKILL.md                        # 本書 §2〜7 の要約(作業ループ・規約・アドレッシング・微調整モード)
├── references/
│   ├── subset-cheatsheet.md        # 語彙早見表(subset-spec.md から生成)
│   └── cli.md                      # CLI リファレンス
└── examples/
    └── prompts.md                  # §6 の指示パターン集
```

- **生成物であって手書きしない。** cheatsheet と SKILL.md の語彙記述は `docs/subset-spec.md` を単一の真実としてビルドで生成する。仕様変更のたびに再生成され、スキルと実装の知識ドリフトを構造的に防ぐ。
- **配布は `deck init` がプロジェクトに同梱する。** 新規デッキプロジェクトの雛形生成時に `.claude/skills/beamer-deck/` として書き込む。リポジトリと一緒に配布されるので、クローンした人の Claude Code がそのまま拾える(ゼロインストール)。生成時に CLI のバージョンを記録し、`deck lint` が CLI と skill の版ずれを警告する。
- SKILL.md 本文は短く保ち、詳細は references/ への参照で段階開示する(スキルの一般作法に従う)。
- 将来: プラグインとしてのユーザーレベル配布、エディタ内蔵チャット(Agent SDK)への同一スキル注入。どちらもプロトコルは本書のまま変わらない。

## 9. 採らなかった案

| 案 | 却下理由 |
|---|---|
| 構造化パッチ API(JSON 操作をエージェントに出させる) | テキスト + 正規形で既に diff が安定しており、共通言語(ソースそのもの)を迂回する機構は思想に反する |
| デッキ操作用 MCP サーバー | ファイル + CLI で足りる。CLI はスキルから叩けるし、人間も同じものを使える。内蔵チャット実装時に必要なら検討 |
| エージェントにプレビュー HTML を検証させる | プレビューは近似であり権威ではない。検証は TeX(check / snapshot)に一本化する |

## 付録: SKILL.md ドラフト

Phase 8 で生成する SKILL.md の骨子(語彙表・CLI 詳細は references/ に分離)。

```markdown
---
name: beamer-deck
description: >
  Beamer サブセット形式のスライドデッキ(deck プロジェクトの .tex)を
  作成・編集するときに使う。スライドの生成・修正・レイアウト検証・
  PDF 書き出しの手順、語彙の制約、フレームの参照方法を含む。
---

# Beamer デッキの編集

このプロジェクトの .tex は Beamer のサブセットで書く。語彙の範囲は
references/subset-cheatsheet.md を参照。範囲外の LaTeX も書けるが、
エディタでの編集性が落ちるため、必要な場合のみ使い、報告に明記する。

## 作業ループ(必ず従う)
1. `deck outline <file>` で構造を把握し、対象フレームの行範囲だけ読む
2. 編集する
3. `deck format <file> --write` → `deck lint <file>` をエラー 0 まで
4. 内容量やレイアウトを変えたら `deck check <file>`(溢れ検出)
5. 見た目に確信がなければ `deck snapshot --frame <addr>` で画像確認
6. 変更フレームのアドレス + 一行説明 + lint/check 結果を報告

## 規約
- ツール管理プリアンブルは編集禁止。マクロ定義は %% macros 領域のみ
- 指示されていない変更(順序・枚数・既存文言)をしない
- 画像は assets/ に置く。新規デッキはまずアウトラインを提案して合意を取る
- git 管理下では 1 指示 = 1 コミット

## フレームの参照
序数(「3枚目」)・タイトル・label のいずれでも受ける。曖昧なら
`deck outline` の一覧を示して確認する。

## 微調整依頼([adjust] ブロック、または文脈上の微調整)
- target のフレーム外を変更しない。必要と判断したら変更せず提案を返す
- 見た目の依頼は snapshot で前後比較 + check。文言のみなら lint だけでよい
- 報告は 1〜2 行に抑える。「もう少し」は直前と同じ target への追加依頼
```
