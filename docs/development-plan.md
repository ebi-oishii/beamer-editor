# 開発計画

ステータス: draft / 最終更新: 2026-07-10

## 方針

1. **最短でドッグフーディングに到達する。** 「AI がファイルを書き、人間がエディタでプレビューを見る」ループ(M2)が成立した時点で、このツール自体の開発資料をこのツールで作り始める。
2. **依存の根本から作る。** すべてのコンポーネントが AST に依存するため、パーサと AST 設計が最初。GUI 操作は要件上「限定的でよい」と合意済みなので後回し。
3. **ゴールデンサンプル駆動。** 実際の発表を模したサンプルデッキを最初に手書きし、全フェーズのテストデータ兼受け入れ基準にする。
4. **シェルは両立て、UI は一度だけ書く。** VS Code 拡張と Electron を同時に持つが、プレビュー・GUI・依頼ボックスは共有パッケージ `ui` に実装し、シェル側は ShellHost アダプタに限定する。ドッグフーディングは先に立ち上がったシェル(見込み: VS Code 拡張)で開始し、もう一方の完成を待たない。

## マイルストーン

| M | 状態 | 到達点 |
|---|---|---|
| M1 | 読める | サンプルデッキがブラウザでスライドとして表示される |
| M2 | 書ける | いずれかのシェルで編集 + 即時プレビューが成立し、外部の AI がファイルを書くと即時反映される(ドッグフーディング開始) |
| M3 | 出せる | PDF 書き出しと部分コンパイルが動き、キメラプレビューが完成する |
| M4 | 触れる | GUI 操作・CLI・エディタからの AI 微調整依頼が揃う |
| M5 | 配れる | Web 版(劣化モード)と両シェルの配布(.vsix / Marketplace、インストーラ) |

## フェーズ詳細

規模感は 1 人で集中した場合の相対値: S = 〜2 日 / M = 〜1 週 / L = 1 週超。

### Phase 0: リポジトリ基盤とゴールデンサンプル(S)

- pnpm workspace モノレポ、TypeScript、vitest、Biome(lint + format)。
- `fixtures/` にゴールデンサンプルデッキを 3 本手書きする:
  1. `basic.tex` — サブセット語彙だけの標準的なデッキ(15 フレーム程度)
  2. `macros.tex` — マクロ定義と呼び出しを多用したデッキ
  3. `kitchen-sink.tex` — サブセット外(TikZ、`\only`、凝ったマクロ)を意図的に混ぜたデッキ
- サンプルは手元の LaTeX 環境(または tectonic)でコンパイルが通ることを確認しておく。
- 完了条件: `pnpm test` が空のテストで通り、サンプル 3 本が PDF になる。

### Phase 1: core — トークナイザ + パーサ + AST(L)【最重要】

- LaTeX トークナイザ(バックスラッシュ命令、`{}`、`[]`、`%`、数式区切り、verbatim 例外)。
- フレーム分割とフレーム単位の再パース。
- サブセット仕様 v1 の再帰下降パース。**未知 → 生ブロック**の 3 段フォールバック(コマンド / 環境 / フレーム)。
- コメントと source span の保持。
- テスト: スナップショットテストを大量に。`kitchen-sink.tex` で生ブロック境界が仕様どおりに切れることを確認。
- 完了条件: サンプル 3 本がパースでき、AST から素朴に再出力したテキストが再パースで同一 AST になる(ラウンドトリップ)。

### Phase 2: core — フォーマッタ + リンター(M)

- 正規形の実装。冪等性テスト(`format(format(x)) == format(x)`)、コメント保持テスト。
- リント規則 L001〜L008(L004 のファイル存在確認は環境非依存にするため、ファイルシステムアクセスを注入可能にする)。
- 完了条件: サンプル 3 本の正規形がレビューで合意され、fixture として固定される。

### Phase 3: core — マクロ展開器(M)

- 定義のパース(展開可能性の判定)、単純置換の展開、ソースマップ。
- 展開不能な呼び出しの生ブロック化。
- Phase 2 と並行可能(どちらも Phase 1 の AST にのみ依存)。
- 完了条件: `macros.tex` の展開結果が期待スナップショットと一致し、任意の展開後位置から元ソース位置を引ける。

### Phase 4: renderer — HTML プレビュー(M)

- AST → HTML。論理サイズ固定 + CSS transform スケール。KaTeX 統合。
- テーマ CSS 1 本(default か metropolis)。
- オーバーレイのステップ表示。生ブロックはこの段階ではプレースホルダ表示。
- 開発用に `apps/web` を簡易ビューア(ファイルドロップ → 表示)として先行させ、動作確認の場にする。
- Phase 2/3 と並行可能。
- 完了条件(= M1): `basic.tex` がブラウザでスライドとして閲覧でき、数式・段組・ブロック・オーバーレイが表示される。

### Phase 5: 共有 UI + シェル 2 本のエディタ MVP

3 つのサブトラックに分かれる。5a を最初に固め、5b / 5c は並行できる。

**5a. packages/ui — 共有 UI と ShellHost 契約(S)**

- ShellHost インターフェースの定義: 編集の適用 / 変更の購読 / ファイルダイアログ / コンパイル呼び出し / AI 依頼の送信。シェル差はここに閉じ込める。
- プレビュー表示(renderer の出力のホスト、フレーム選択、ステップスライダー)を React で実装。

**5b. apps/vscode — VS Code 拡張シェル(S)**

- プレビュー WebviewPanel(`ui` をホスト)。ドキュメント変更イベント → 再パース → 更新。
- lint は DiagnosticCollection(Problems パネル・波線)で表示。
- ステータスバーに現在フレームのアドレス(序数 / label)、クリックでコピー([ai-protocol.md](ai-protocol.md) §3)。
- 外部(AI)編集: バッファ未編集時の再読込は標準機能。開いているドキュメントへは WorkspaceEdit として適用し undo 統合する。
- エディタ・保存・競合処理・git diff はプラットフォーム標準をそのまま使う(実装しない)。

**5c. apps/desktop — Electron シェル(M)**

- electron-vite。左 CodeMirror 6 / 右プレビュー(同じ `ui` をホスト)の 2 ペイン。
- ファイル open / save、chokidar 監視(未編集バッファは黙って再読込、編集中は通知)。
- CodeMirror に lint 指摘のインライン表示、ステータスバーのアドレス表示。
- 外部(AI)編集の取り込みを undo 履歴の 1 ステップとして扱い、変更を通知する(AI の微調整を Cmd+Z で戻せる)。

完了条件(= M2): 先に立ち上がったシェルで `basic.tex` を編集しながらプレビューが追従し、別プロセス(例: Claude Code)の書き込みが即時反映される。**ここからドッグフーディング開始。** もう一方のシェルが同条件を満たした時点でパリティ達成とする。

### Phase 6: compiler — 書き出しと部分コンパイル(M)

- tectonic 呼び出し(まず開発機のバイナリをパス指定。配布方式はこのフェーズで決定)。
- PDF 書き出し(展開前ソースをそのままコンパイル)+ 書き出し PDF のプレビューペイン表示切り替え。
- 生ブロックの standalone 部分コンパイル → pdf.js ラスタライズ → コンテンツハッシュキャッシュ → プレビューに差し込み。
- フレーム単位の画像出力と、コンパイルログの Overfull 警告をフレームアドレスへ割り付ける機構(Phase 8 の `deck snapshot` / `deck check` の土台)。
- 完了条件(= M3): `kitchen-sink.tex` の TikZ が(初回コンパイル後)プレビューに画像として表示され、PDF 書き出しが 1 クリックで動く。

### Phase 7: GUI 操作(M)

- スライド一覧ペイン(サムネイル): 並べ替え・複製・削除・挿入。
- フレームタイトルと `\item` のインライン編集。画像差し替え。
- すべて AST 変換 + 触れたフレームの正規形再出力。undo/redo はソース履歴に統合。
- 実装は `ui` に 1 回だけ書き、編集適用は ShellHost 経由(vscode は WorkspaceEdit、desktop はエディタバッファ更新)。受け入れは両シェルで確認する。
- 完了条件: GUI 操作の結果がソースペイン(エディタ)に即座に現れ、undo がテキスト編集と同一スタックで効く。両シェルで同一チェックリストが通る。

### Phase 8: cli — CLI と AI プロトコル整備(M)

[ai-protocol.md](ai-protocol.md) の実装フェーズ。

- CLI の完全なセット: `deck outline` / `deck lint` / `deck format` / `deck check` / `deck snapshot` / `deck export`(各 `--json` 対応。check / snapshot は Phase 6 の機構を使う)。
- SKILL.md と `references/subset-cheatsheet.md` を `docs/subset-spec.md` から**ビルドで生成**する仕組み。`deck init` が新規デッキプロジェクトに `.claude/skills/beamer-deck/` として同梱する(版ずれは L010 で警告)。
- 指示パターン集(examples/prompts.md)。このリポジトリ自身にもスキルを配置し、資料作成で運用検証する。
- 完了条件: AI に「アウトライン提案 → 合意 → 生成 → lint/check 通過」の流れで新規デッキを作らせ、人間がエディタで微調整して PDF 書き出しまでの一連が実演できる。

### Phase 9: エディタからの微調整依頼(M)

[ai-protocol.md](ai-protocol.md) §7 のエディタ統合。Phase 5 と Phase 8 に依存する(Phase 7 の GUI とは独立)。

- プレビュー・スライド一覧・ソースの選択から「AI に依頼」→ 依頼ボックス → `[adjust]` メッセージ(target アドレス + 選択ソースの引用 + 指示文)を合成。依頼ボックスは `ui` に実装し、セッションの起動・維持は ShellHost アダプタが担う。
- デッキごとに 1 本の永続エージェントセッション(claude CLI の resume / Agent SDK)を保ち、そこへ送信。「もう少し」が通じる文脈を維持する。仕組みは両シェルで共通(どちらも Node からのプロセス起動)。
- 反映はファイル監視経由(Phase 5 の undo 統合がそのまま安全弁になる)。
- 完了条件(= M4): プレビューで表を選択して「窮屈なので詰めて」と依頼すると、snapshot/check 検証つきの修正がファイル経由で反映され、Cmd+Z で戻せる。両シェルで成立すること。

### Phase 10: web 版と配布(M)

- `apps/web` を正式化(生ブロックはプレースホルダ、書き出し非対応の劣化モード)。
- VS Code 拡張のパッケージング(.vsix 配布、必要なら Marketplace 公開)。
- Electron の配布パッケージング(インストーラ、macOS 公証、tectonic 同梱 or 初回ダウンロード)。
- 必要になった時点でコンパイル API サーバー(第二段階)を検討。
- 完了条件(= M5): チームメンバーが .vsix またはインストーラから導入して使える。

## 並行作業の目安

チームで分担する場合、クリティカルパスは Phase 0 → 1 → (2 | 3 | 4 並行) → 5a → 5b → 6。

- Phase 1 完了後: フォーマッタ / マクロ展開 / レンダラの 3 本は独立に進められる(AST の型だけ合意すればよい)。
- Phase 5a(ShellHost 契約)を固めれば、5b(vscode)と 5c(desktop)は別担当で並行できる。ドッグフーディングは先に完成した方(見込み: 5b)で始める。
- テーマ CSS の作り込みとサンプルデッキの拡充は、どのフェーズとも並行できる。
- Phase 7(GUI)と Phase 8(CLI)は独立。Phase 9(微調整依頼)は Phase 5 と 8 の完了後に着手できる。

## テスト戦略

- core はスナップショット中心(パース結果・整形結果・展開結果)。冪等性とラウンドトリップは property 的に全 fixture へ一括適用する。
- renderer は fixture ごとの HTML スナップショット + 主要スライドの目視確認ページ。
- `ui` は ShellHost をモックしたコンポーネントテスト(GUI 操作が正しい編集適用の呼び出しに変換されるかを検証)。シェルアダプタ自体は薄く保ち、手動のパリティチェックリストで補う。
- compiler は tectonic 依存のため CI ではキャッシュ・キュー・ハッシュのロジックのみ単体テストし、実コンパイルはローカルの統合テストとする。
- 「AI が書いたデッキが lint を通る率」を M4 以降の品質指標として fixture に蓄積する。

## 最初の一歩(着手チェックリスト)

1. Phase 0 のモノレポ雛形(git リポジトリは docs 先行で初期化済み)
2. ゴールデンサンプル 3 本の執筆(サブセット仕様の妥当性検証を兼ねる — 書いてみて窮屈な箇所は仕様へフィードバック)
3. AST の型定義ドラフト(`packages/core/src/ast.ts`)をレビューして合意
4. Phase 1 のトークナイザから実装開始
