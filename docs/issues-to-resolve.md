ステータス: レビュー結果 / 作成日: 2026-07-10
対象: `additional-requirements.md` を加えた要件定義一式(design / subset-spec / ai-protocol / development-plan)

## この文書の位置づけ

追加要件を含めた全体をレビューした結果、**要件定義として致命的な矛盾はなく、次段階に進める水準**にある。追加要件はむしろ全体を現実的に絞り込んでいる(シェル二重化の撤回、GUI のキャンバス限定)。

ただし「追加要件が既存4文書を上書きしているのに既存文書側が旧決定のまま残っている」ドリフトが複数ある。SKILL.md / cheatsheet を `subset-spec.md` から**ビルド生成**する設計である以上、仕様の食い違いはそのままエージェントの知識ドリフトになる。実装着手前に潰す必要がある。

区分は次の3つ。

- **A. 文書リコンサイル** — 判断は済んでいる。既存文書の記述を追加要件に合わせて直すだけ(機械的)。
- **B. 実装可能性の修正** — 計画の順序・見積もりに実行不能または過小評価がある。
- **C. 設計判断が必要な穴** — 追加要件で新たに生じた、まだ決めていない論点。着手前に合意が要る。

優先度の考え方: **C-1(本文領域の定義)が他のすべてに波及するため最優先**。次に A の一括修正、その後 C の残りと B。

## C. 設計判断が必要な穴(着手前に合意)

### C-1【最優先・全体に波及】「本文領域」の定義

追加要件 §3.4 は座標原点を「フレーム本文領域の左上」とする。しかし Beamer のフレームタイトル高さはタイトル内容(1行 / 2行)で変わり、本文領域の上端が動く。**座標の意味そのものが揺れる。** TeX 側マクロと HTML レンダラの両方がこの定義に依存するため、他のどの判断より先に決める必要がある。

決めるべき選択肢:

- **案①(推奨)**: タイトル領域を固定高(1行相当)と定義し、2行以上のタイトルは lint 警告。→ 座標の意味が安定する。
- **案②**: 原点をスライド全体の左上に取り、タイトルは別枠扱い。

いずれも default テーマの実寸を測ってから最終決定する。**この決定なしに Phase 0.5 に入れない。**

### C-2 キャンバスフレームで `deck check` の安全網が効かない

現行設計の柱は「エージェントが `deck check` で Overfull を先回りして潰す」ことだが、textpos 系の絶対配置は**縦にはみ出してもスライド外に静かに消えるだけで Overfull 警告が出ない**(箱内の hbox 溢れは出るが、縦溢れ・オブジェクト重なりは検出不能)。canvas フレームでは check の価値が落ち、snapshot(視覚検証)頼みになる。

対策案: `\pdfsavepos` / `zref-savepos` で各オブジェクトの実測位置・高さを aux に吐かせ、`deck check` が canvas フレームについて「本文領域外へのはみ出し」「オブジェクト重なり」を機械検出する。→ これを **L012 の実装方式**として要件に書き足す。

### C-3 `decktext` 内部の許容語彙

§3.2 は「単純な箇条書き」、AST は `children: BlockNode[]` と広い。itemize は何段まで許すか、block 環境や数式ディスプレイは可か。L014 は `deckcanvas` 直下しか見ていないため、**decktext 内部の許容語彙とその lint** を別途定義する必要がある。

### C-4 canvas オブジェクトとオーバーレイの関係

`\pause` や `<n->` を `decktext` に付けられるのか、v1 対象外なのかが未記載。対象外とするなら明記し lint を足す。

### C-5 画像の縦横比取得の仕組み

高さ自動算出・L015・GUI の比率保持スケールはすべて画像寸法の取得を前提とするが、**誰が寸法を取るか**(特に PDF 画像の MediaBox 読み)が未定義。L004 と同じ「ファイルアクセス注入」パターンで core に寸法プローブを注入する形を決めておく。

### C-6 `deck*` 名前空間の予約

`deckcanvas` 等はツール管理プリアンブルで定義されるが、ユーザーが `%% macros` 領域で `\renewcommand{\deckimage}` した場合の扱いが未定義。予約名前空間として lint(**L016 相当**)を足す。

### C-7 競合検査の frame 同一性

§4.6 は「対象 frame の label と内容ハッシュ」で競合検知するが、label は canvas フレームにしか必須化されていない(L011)。**無ラベルフレームへの adjust 依頼時の同一性**(序数は並べ替えで壊れる)をどう取るか。案: adjust 対象になった時点で label を自動付与 / ソースハッシュ + 近傍探索。

### C-8 小さいもの(まとめて確認)

- `deck-source-version` 不一致の lint ID が未採番。
- ドラッグ中プレビューは「ソースが唯一の真実」原則の例外(UI 一時状態のオーバーレイ)になる。renderer / ui の契約として明文化する。
- 16:9 限定なのに subset-spec は `aspectratio=43` を許す。「4:3 デッキ内の `deckcanvas` は lint 警告」を足すか決める。
